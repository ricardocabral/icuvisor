# v0.5 internal beta exit interview

Run this after the participant completes the beta window or withdraws. Keep notes redacted and map them to [measurement](measurement.md) fields.

## Opening

> Thank you for trying icuvisor. Please avoid sharing API keys, athlete IDs, raw workout values, full transcripts, or screenshots with data. I am looking for workflow feedback, blockers, and whether this should continue toward v1.0.

## Questions

1. How long did it feel like it took from install to the first useful tool call, and where did time feel wasted? (KR1)
2. Which 3-5 icuvisor tool calls or prompts were most useful by name or description only?
3. What was the first moment where the product felt trustworthy or untrustworthy?
4. Would you recommend icuvisor to a similar intervals.icu user today? Why or why not? (PRD §7.4 #6)
5. What would block you from using it for daily training questions?
6. Did you need mobile, menu-bar, tray, or remote access for this to fit your routine? (PRD §7.4 #8)
7. If you saw schema-change/catalog-hash guidance or had to start a new conversation, was that clear? (PRD §7.4 #12)
8. Were the setup docs for Claude Desktop or Claude Code complete enough without maintainer improvisation?
9. Did `icuvisor diagnostics` provide enough support context while staying safely redacted?
10. For coach-mode testers only: did roster selection, per-athlete ACLs, and catalog-cache behavior make sense without exposing roster names or athlete IDs?
11. What surprised you, positively or negatively, about asking an AI assistant to use intervals.icu data?
12. If you could change one thing before v1.0, what would it be?

## Close

- Confirm whether the participant consents to anonymized quotes in aggregated findings.
- Ask whether any blockers should be filed as GitHub issues.
- Remind them they can revoke participation by replying "withdraw" before findings are published.
